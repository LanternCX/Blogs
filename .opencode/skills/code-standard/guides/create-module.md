# 创建模块

## 目标

在新增模块时, 先定义单一职责、公开接口与依赖边界, 再落地实现, 避免一开始就形成大而杂的聚合模块。

## 操作步骤

1. 先写清模块只负责什么, 不负责什么
2. 到 `references/architecture-rules.md` 确认模块边界、接口与依赖方向
3. 把配置、存储、日志等横切能力通过明确入口注入, 不在模块内部随意散落创建
4. 涉及持久化时, 到 `references/persistence-rules.md` 确认程序运行目录下的落盘策略
5. 涉及日志时, 到 `references/logging-rules.md` 确认 stdout 与滚动落盘的双通道要求

## 通过标准

- 模块名称与目录结构能体现职责边界
- 模块接口少而稳定, 调用方不需要理解内部细节
- 模块内部变化不会牵动多个无关模块联动修改
- 持久化与日志方案已经分别对齐对应 `references/` 文档

## 常见反模式

- 以工具箱、common、misc 等模糊名称承载多种无关职责
- 在模块内部绕过 `references/persistence-rules.md`, 自己发明落盘路径规则
- 先写大量实现, 之后再回头补边界, 导致职责已经耦死
