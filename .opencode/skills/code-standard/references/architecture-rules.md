# 架构规则

## 核心原则

- 高内聚: 每个模块只围绕一个稳定职责组织代码
- 低耦合: 模块之间只通过最小必要接口协作
- 依赖单向: 上层可以依赖下层抽象, 下层不能反向感知上层编排
- 细节内聚: 易变实现收在模块内部, 不向外扩散

## 模块边界规则

- 一个模块内的函数、类型、文件必须共同服务于同一职责
- 发现一个模块同时承担流程编排、数据落盘、日志拼装、外部调用等多类职责时, 应拆分
- 公共模块只沉淀稳定复用能力, 不承载场景特化逻辑
- 禁止使用模糊目录承载未分类实现, 例如 `misc`、`tmp`、`helper` 作为长期归宿

## 依赖规则

- 调用方依赖接口语义, 不依赖被调方内部状态布局
- 禁止跨层绕过正式入口直接访问底层细节
- 禁止循环依赖、双向回调绑定与隐式全局共享状态
- 横切能力通过明确组件统一提供, 不在每个业务模块里重复造轮子

## 设计信号

- 改一个需求却需要同时修改多个不相关模块, 说明耦合过高
- 模块名称无法用一句话说明职责, 说明边界不清
- 新增功能必须复制旧模块大段实现, 说明抽象位置不对
