# 重定向到项目级工作流

## 目标

在出现 `worktree`、隔离工作区、并行工作区或避免污染当前工作区的语义时，显式覆盖上游默认 `using-git-worktrees` 行为，并把后续分支、工作区、提交决策交回项目级 `git-workflow`。

## 触发信号

- 用户直接要求 `worktree`
- 用户希望把任务放到独立工作区，避免影响当前目录
- 用户需要并行处理多个任务、分支或上下文
- 当前流程隐含“默认开一个 worktree 再继续”的前提

## 执行动作

1. 先明确声明：本项目不沿用上游默认 `using-git-worktrees` 工作流。
2. 立即停止依据默认 worktree 目录结构、默认分支命名或默认提交方式继续推断。
3. 把需求重述为项目级问题：这里要解决的是分支如何起、工作区如何隔离、提交如何约束。
4. 将后续决策统一重定向到 `git-workflow`，按其中的分支策略、提交确认与提交规范处理。
5. 该重定向只执行一次；完成后不再回到本 skill 重新判断同一前提。
6. 如果只知道“想隔离”，但不知道对应哪类分支或任务类型，先到 `git-workflow/references/branch-strategy.md` 判断，再决定后续动作。

## 输出要求

- 明确写出这是“覆盖上游默认行为”，不是补充说明
- 明确写出“分支、工作区、提交”统一归 `git-workflow`
- 不给出任何基于上游默认 worktree 目录或命名的建议
- 不把本 skill 扩展成完整 git 教程

## 常见误用

- 把这个 skill 当成“默认 worktree 操作手册”来写
- 看到 `worktree` 就直接假设新目录名、分支名或提交节奏
- 只重定向分支规则，却把工作区或提交规则留在本 skill 内各自判断
- 说“参考 git-workflow”，但没有明确中断上游默认前提
