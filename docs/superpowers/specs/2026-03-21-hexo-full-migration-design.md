# Hexo 全量迁移设计说明

## 概述

本文档定义本仓库第二阶段的博客迁移设计, 目标是在已完成 Hexo 基础骨架与抽样迁移验证的前提下, 以“内容源目录 + Hexo 生成目录”双层结构重组现有博客内容。

本阶段聚焦 4 件事:

- 重组现有博客内容目录, 形成统一的内容源根目录
- 在保留分类语义的前提下, 将内容源批量转换为 Hexo 可消费文件
- 产出完整迁移报告与人工复核清单
- 让 Hexo 根目录只保留站点配置、主题、脚本与生成产物所需结构

## 目标

- 将现有博客文章与页面重组到统一内容源目录
- 保留目录表达的分类语义, 但不强求保留当前根目录物理布局
- 让 Hexo 根目录更贴近推荐最佳实践
- 让 Hexo 最终消费的内容继续落在 `source/_posts/` 与页面目录
- 对异常文件给出人工复核清单, 而不是静默跳过

## 非目标

- 本阶段不进行主题视觉还原或 Akina 风格迁移
- 本阶段不接入 Memos、Telegram Bot、CI/CD 或部署流程
- 本阶段不尝试自动修复所有异常 Markdown 结构
- 本阶段不将 `.marp.md` 文件迁入站点
- 本阶段不把开发与运维文档混入博客站点输出

## 方案对比

### 方案 A: 内容源目录 + 生成到 Hexo

做法:

- 将现有博客内容先重组到统一内容源目录, 如 `content/`
- 再由迁移脚本把 `content/` 下的文章和页面生成到 Hexo `source/` 目录

优点:

- Hexo 根目录干净, 角色边界清晰
- 分类语义保留, 但物理目录可以重组
- 后续做全量迁移、草稿、静态资源和 CI/CD 更容易维护

缺点:

- 相比直接写入 `source/`, 多了一层内容源到站点内容的同步过程

### 方案 B: 直接将现有目录改造成 Hexo source 子树

做法:

- 直接把现有目录结构并入 Hexo `source/` 体系
- 让 Hexo 尽量直接读取迁移后的目录

优点:

- 少一层生成步骤

缺点:

- Hexo 内容模型与原始内容模型容易耦合
- 后续目录治理和脚本边界会更混乱

### 方案 C: 保留原始目录为唯一事实源, 增量同步到 Hexo

做法:

- 尽量不调整原有目录, 仅编写同步脚本产出 Hexo 内容

优点:

- 前期目录变更更少

缺点:

- Hexo 根目录与原始内容目录会长期共存并相互牵制
- 不利于后续形成整洁的项目结构

## 推荐方案

采用方案 A。

原因:

- 用户的核心诉求是保留分类语义, 而不是保留当前根目录物理布局
- 该方案最接近“内容源”和“站点消费层”分离的最佳实践
- 第一阶段已证明 Hexo 依赖链路与渲染链路可行, 第二阶段更应整理结构, 而不是继续在仓库根目录堆叠内容目录

## 目录模型设计

### 新的顶层职责划分

本阶段目标是形成以下职责边界:

- Hexo 根目录: 配置、依赖、主题、脚本、测试、生成入口
- 内容源目录: 保存博客文章与页面的原始 Markdown 内容
- Hexo `source/` 目录: 保存供站点实际渲染的生成结果

### 内容源目录

建议引入统一内容源目录 `content/`。

重组后的博客内容按语义收拢到 `content/` 下, 例如:

- `content/ACM-ICPC/`
- `content/Agent/`
- `content/Debug-Lab/`
- `content/EDC/`
- `content/Note/`
- `content/Other/`
- `content/Photography/`
- `content/Robotic/`
- `content/Smart Car/`
- `content/Pages/`

这里的目录层级保留分类语义, 但具体物理位置从仓库根目录迁移到 `content/` 下。

### Hexo 消费目录

- 普通文章统一生成到 `source/_posts/`
- 独立页面统一生成到 `source/<page>/index.md`

Hexo 最终只消费 `source/` 下的生成结果, 不直接把 `content/` 当作渲染输入。

## 内容判定规则

### 站点内容范围

只迁移博客内容。

本阶段纳入迁移的博客内容目录默认包括:

- `content/ACM-ICPC/`
- `content/Agent/`
- `content/Debug-Lab/`
- `content/EDC/`
- `content/Note/`
- `content/Other/`
- `content/Photography/`
- `content/Robotic/`
- `content/Smart Car/`
- `content/Pages/`

站点内容判定采用“目录白名单 + 文件规则”双重约束:

- 只有位于上述白名单目录下的 Markdown 文件才进入候选集合
- 候选文件必须满足 `*.md`
- 候选文件若匹配 `*.marp.md`, 直接排除
- 候选文件若文件名为 `README.md`, 默认排除
- 仅当某个 `README.md` 被显式登记到 `.migration/include-readmes.json` 时, 才允许作为博客正文迁移

若后续需要纳入特殊 README, 应通过显式白名单处理, 而不是修改默认规则。

`.migration/include-readmes.json` 采用以下固定格式:

```json
{
  "paths": [
    "content/SomeCategory/README.md"
  ]
}
```

规则说明:

- 路径必须是仓库根目录相对路径
- 路径分隔符统一使用 `/`
- 采用大小写敏感的精确匹配
- 不允许目录级通配或 glob

非法输入处理口径:

- 文件不存在时, 按空白名单处理
- 顶层结构不是对象时, 全量迁移命令直接失败
- 缺少 `paths` 字段时, 全量迁移命令直接失败
- `paths` 不是数组时, 全量迁移命令直接失败
- 数组元素不是字符串时, 全量迁移命令直接失败
- JSON 非法时, 全量迁移命令直接失败
- 路径重复时, 全量迁移命令直接失败
- 路径不是 `README.md` 时, 全量迁移命令直接失败
- 路径不在博客目录白名单下时, 全量迁移命令直接失败
- 路径指向的文件不存在时, 全量迁移命令直接失败
- 路径含前导 `./`、反斜杠或未规范化段时, 全量迁移命令直接失败

### 排除规则

以下内容不进入站点:

- 所有 `.marp.md` 文件
- 开发与运维文档目录, 如 `.opencode/`、`docs/`、`.progress/`
- 所有未落在博客目录白名单中的 Markdown 文件

排除规则以“目录白名单 + 文件规则”为唯一执行口径, 不再引入额外的主观判断步骤。

### 页面与文章分流规则

- 位于 `content/Pages/` 下的单层页面文件默认判定为独立页面
- 其他纳入迁移范围的 Markdown 默认判定为普通文章
- 页面不写 `categories`
- 文章的 `categories` 取 `content/` 下的一级目录名
- `content/Pages/README.md` 即使进入 README 白名单, 也不作为页面迁移, 仍按排除项处理

分流规则以目录位置为唯一执行口径:

- `content/Pages/<name>.md` -> `page`
- 除上述路径外的其他候选 Markdown -> `post`
- 本阶段不引入额外的页面白名单或启发式判定

## Front Matter 规则

普通文章至少包含:

- `title`
- `date`
- `updated`
- `categories`

独立页面至少包含:

- `title`
- `date`
- `updated`

字段来源与第一阶段保持一致:

- `title`: 取正文中的第一个 Markdown 标题
- `date`: 取文件历史中的首次 author date
- `updated`: 取文件历史中的最后一次 author date
- `categories`: 取 `content/` 下的一级目录名

## 路径与命名规则

### 内容源路径

- 原始博客内容从仓库根目录迁移到 `content/` 下
- 目录语义保留, 但根目录不再直接承担内容分类职责

### 文章路径

- Hexo 目标文章文件默认生成到 `source/_posts/`
- 目标文件名默认沿用原 Markdown 文件名
- 若两个或更多源文件映射到相同目标路径, 视为目标路径冲突, 进入人工复核清单
- 若目标文件名为空、包含路径分隔符、为 `.` 或 `..`, 视为非法目标文件名, 进入人工复核清单

### 页面路径

- `content/Pages/about.md` -> `source/about/index.md`
- `content/Pages/friends.md` -> `source/friends/index.md`
- 其他页面文件按页面文件名本体映射到 `source/<slug>/index.md`

页面路径补充规则:

- 仅处理 `content/Pages/` 下的单层页面文件
- 本阶段不支持 `content/Pages/` 下的嵌套页面目录
- 页面 `slug` 默认取文件名本体
- 若页面 `slug` 为空, 或两个页面映射到相同目标路径, 则进入人工复核清单, 不静默写入

`content/Pages/` 边界补充规则:

- 若 `content/Pages/` 下出现子目录, 默认不自动迁移为页面, 直接进入人工复核清单
- 若页面文件存在同名资源目录, 资源目录不直接并入页面输出, 需按静态资源迁移规则处理
- `content/Pages/` 之外的候选 Markdown 即使历史上曾被当作页面使用, 本阶段也统一按 `post` 处理

## 标题识别口径

本阶段的 `title` 识别采用与第一阶段一致的 ATX 标题口径:

- 只识别正文中的 Markdown ATX 标题, 即 `#` 到 `######`
- 从正文自上而下取第一个可识别标题
- 不将 front matter 中的字段值视为正文标题
- 不从代码块中提取标题
- 若找不到可识别标题, 则视为“缺少可识别标题”, 进入人工复核清单

## 静态资源迁移策略

全量迁移不仅处理 Markdown, 还必须处理文章与页面引用的本地静态资源。

### 资源范围

本阶段关注以下可识别的本地资源:

- Markdown 正文中以相对路径显式引用的图片资源
- Markdown 正文中以相对路径显式引用的附件文件
- front matter 中显式声明且指向相对路径的资源字段, 仅限以下字段名: `cover`, `banner`, `thumbnail`, `image`

本阶段不自动迁移“仅与文章同目录但未被正文或 front matter 显式引用”的旁挂文件。此类文件若需要进入站点, 应进入人工复核清单。

### 迁移规则

- 若文章或页面引用的是外链资源, 不做迁移
- 若引用的是相对路径本地资源, 需将资源复制到 Hexo 可访问位置
- 文章资源统一生成到与目标文章同名的资源目录, 例如 `source/_posts/foo.md` 对应 `source/_posts/foo/`
- 页面资源统一生成到页面目录下的资源位置, 例如 `source/about/index.md` 对应 `source/about/`
- 若资源路径无法解析、资源缺失或资源重名冲突, 进入人工复核清单

### 链接处理

- 若资源目标路径发生变化, 需同步改写 Markdown 中的相对链接
- 资源链接改写必须可重复执行, 避免多次迁移产生重复改写
- 若资源引用不属于 Markdown 正文相对路径链接或 front matter 允许字段集合, 则不做自动改写, 并进入人工复核清单

## Git 时间提取策略

由于本阶段会先重组物理目录, 必须明确 `date` 与 `updated` 的提取时机。

- Git 时间提取应基于“重组前的原始路径”完成, 不能依赖重组后的新路径重新推断
- 目录重组应优先使用 `git mv`, 以最大限度保留历史关联
- front matter 生成使用重组前提取到的首次/最后 author date 结果
- 本阶段默认不依赖 `--follow` 作为主要策略, 仅在人工排查时作为辅助验证手段
- 若 `git log --format=%aI -- "<relative-path>"` 返回空结果或命令失败, 视为时间无法可靠提取, 文件进入人工复核清单, 不伪造时间

## 迁移流程设计

### 总体流程

1. 将现有博客内容目录收拢到 `content/`
2. 在重组前或重组过程中提取原始路径的 Git 时间元数据
3. 扫描 `content/` 下的候选 Markdown 文件
4. 按规则排除 `.marp.md` 与非博客内容
5. 对每个文件执行标题提取、页面/文章分流、目标路径映射与资源分析
6. 清理并重建 Hexo `source/` 中由迁移流程负责的内容输出
7. 生成 Hexo 目标文件到 `source/`
8. 生成全量迁移报告与人工复核清单
9. 验证首页、分类、归档、页面、文章详情页是否正常

### 结构拆分

为保持边界清晰, 本阶段应拆分以下职责:

- 目录重组器: 负责将现有博客内容移动到 `content/`
- Git 元数据采集器: 负责在重组前记录原始路径的时间信息
- 扫描器: 枚举 `content/` 下候选 Markdown 文件并执行过滤规则
- 迁移器: 复用现有规则模块, 为单个文件生成 front matter 和目标路径
- 资源处理器: 负责复制本地静态资源并改写引用路径
- 写入器: 将迁移结果写入 Hexo `source/`
- 报告生成器: 输出迁移清单与人工复核项

### `source/` 清理与覆盖策略

为保证全量迁移可重复执行, 必须明确 Hexo 消费目录的覆盖规则。

- 迁移流程负责管理的输出范围包括 `source/_posts/` 与 `.migration/pages-manifest.json` 中登记的页面目标目录
- 在每次全量迁移前, 应清理上述由迁移流程托管的旧输出, 再重新生成
- 不应无差别删除 `source/` 下所有内容, 以避免误删非迁移流程管理的站点文件
- 清理范围、保留范围与最终生成范围都必须可被测试验证

`.migration/pages-manifest.json` 用于记录由迁移流程生成的页面目录清单, 使重复执行时可以安全清理旧页面输出而不影响手工维护文件。

## 异常内容处理

以下情况不阻塞整体迁移, 但必须进入人工复核清单:

- 缺少可识别标题的 Markdown
- 缺少 Git 历史或无法可靠提取时间的文件
- 文件名冲突或目标路径冲突
- 资源路径无法解析、资源缺失或资源复制冲突
- 因目录重组后不在博客目录白名单中的文件

人工复核清单必须显式说明:

- 来源路径
- 问题原因
- 是否已生成降级版本, 或已跳过写入

## 迁移报告格式

### 输出路径

- 全量迁移报告固定输出到 `docs/migration/hexo-full-migration-report.md`
- 人工复核清单作为该报告中的独立章节, 不另起第二个文件
- 迁移流程生成的页面目录清单输出到 `.migration/pages-manifest.json`

### 最小字段

全量迁移报告至少包含以下章节:

- `## 迁移摘要`
- `## 成功迁移`
- `## 降级迁移`
- `## 人工复核未写入`
- `## 统计信息`

每条记录至少包含以下字段:

- 来源路径
- 目标路径
- 内容类型 `post` / `page`
- 分类或页面 slug
- 结果分类
- 备注或问题原因

其中“人工复核未写入”章节的每条记录必须固定包含以下字段:

- 来源路径
- 预期目标路径或目标类别
- 问题原因
- 建议处理方式
- 当前状态 `未写入` 或 `需人工确认后重试`

其中“降级迁移”章节的每条记录必须固定包含以下字段:

- 来源路径
- 实际目标路径
- 降级原因
- 缺失字段或异常类型
- 建议后续动作
- 当前状态 `已写入待复核`

统计信息至少包含:

- 扫描文件总数
- 排除文件总数
- 成功迁移数量
- 降级迁移数量
- 人工复核未写入数量

### 迁移结果分类

本阶段的输出结果分为 3 类:

- 成功迁移: 已写入 Hexo 目标路径, 且元数据完整
- 降级迁移: 已写入 Hexo 目标路径, 但存在缺失字段或资源改写失败等非阻断问题, 并同步进入人工复核清单
- 人工复核未写入: 因标题缺失、路径冲突或其他关键问题未写入 Hexo, 仅记录在人工复核清单中

## 测试与验证策略

本阶段继续采用 TDD。

### 测试重点

- 目录重组规则是否保留分类语义
- `content/` 扫描与过滤规则
- `.marp.md` 排除逻辑
- 页面与文章分流规则在 `content/` 目录模型下是否正确
- 冲突检测与人工复核输出
- 全量迁移后的首页、分类、归档、页面、文章详情页渲染闭环
- 静态资源复制与链接改写规则
- `source/` 清理与重复执行行为

### 站点级验证

至少验证以下命令链路:

- `npm test`
- 本阶段定义的全量迁移命令
- `npm run hexo:clean`
- `npm run hexo:generate`
- `npm run hexo:serve`

### 内容级验证

至少覆盖:

- 首页能列出迁移后的文章
- 分类页可访问
- 归档页可访问
- `about`、`friends` 等页面可访问
- 抽查来自不同分类目录的多篇文章详情页
- 人工复核清单与实际异常文件保持一致

## 成功标准

当以下条件成立时, 可认为本阶段完成:

- 博客内容已从仓库根目录重组到 `content/`
- 分类语义在 `content/` 目录下得到保留
- Hexo 根目录只保留配置、主题、脚本、测试与生成所需结构
