# Hexo 基础骨架设计说明

## 概述

本文档定义本仓库第一阶段的博客迁移设计, 目标是在保留现有内容语义的前提下, 将仓库迁移到符合 Hexo 规范的基础站点结构, 并完成最小可运行闭环。

本阶段聚焦 3 件事:

- 建立标准 Hexo 站点骨架
- 定义旧内容到 Hexo 内容模型的迁移规则
- 通过抽样迁移验证首页、文章页、独立页面能够正常工作

## 目标

- 建立一个可本地运行的 Hexo 基础站点
- 将现有 Markdown 内容映射到 Hexo 标准目录与 front matter 模型
- 保留现有栏目语义, 但不强制保留现有物理目录结构
- 为后续主题迁移、视觉还原、CI/CD 与动态流能力提供稳定基础

## 非目标

- 本阶段不实现 Memos 动态流接入
- 本阶段不实现 Telegram Bot 发布入口
- 本阶段不实现完整 CI/CD 与服务器部署流程
- 本阶段不追求现有站点视觉 100% 还原
- 本阶段不一次性完成全部历史文章的最终清洗

## 方案对比

### 方案 A: 标准 Hexo 初始化 + 渐进迁移

做法:

- 先建立标准 Hexo 目录结构
- 再定义迁移规则, 将现有内容逐步迁入新结构
- 先抽样迁移验证, 后续再扩大到全量迁移

优点:

- 最符合 Hexo 默认模型
- 后续主题、插件、部署流程最容易接入
- 迁移风险可控, 返工成本低

缺点:

- 前期需要接受物理目录结构调整

### 方案 B: 保留现仓库布局, 通过兼容层适配 Hexo

做法:

- 尽量保留现有 Markdown 的路径布局
- 依赖脚本、路由配置或模板兼容去适配 Hexo

优点:

- 初期目录变化较小

缺点:

- 长期维护成本更高
- 后续主题和插件兼容性更差

### 方案 C: 一次性重构到最终形态

做法:

- 初次落地就同时完成站点骨架、主题接入、内容全量迁移与视觉适配

优点:

- 最终结果看起来完整

缺点:

- 风险最高
- 对当前阶段的认知与验证不友好

## 推荐方案

采用方案 A。

原因:

- 用户当前尚未实际使用过 Hexo, 优先建立可运行认知闭环更重要
- 第一阶段应先稳定站点结构与内容模型, 而不是把视觉还原与历史内容清洗同时耦合进去
- 该方案最有利于后续接入 Akina 主题、`typewriterjs`、自定义 CSS 以及 CI/CD

## 内容模型设计

### 最终目录模型

站点采用标准 Hexo 目录作为长期结构:

- `source/_posts/` 存放普通博客文章
- `source/about/index.md`、`source/friends/index.md` 这类目录式页面存放独立页面
- 主题与模板逻辑遵循 Hexo 的主题机制

### 旧内容映射规则

现有仓库中的一级目录语义保留, 但改由元数据表达, 而不是继续作为长期物理结构:

- `ACM-ICPC/`、`Note/`、`Robotic/` 等栏目映射为文章的 `categories`
- `Pages/about.md`、`Pages/friends.md` 等独立页面映射为 Hexo 页面
- 原始 Markdown 目录在迁移初期作为输入来源保留, 不直接作为最终站点结构

### Front Matter 生成规则

迁移后的文章至少包含以下字段:

- `title`
- `date`
- `updated`
- `categories`

字段来源如下:

- `title`: 取 Markdown 正文中的第一个标题
- `date`: 取该文件在 Git 历史中的首次提交时间
- `updated`: 取该文件在 Git 历史中的最后一次提交时间
- `categories`: 取旧目录语义映射结果

这样可以在不要求手工重写全部文章元数据的前提下, 让 Hexo 稳定识别内容。

## 页面模型设计

本阶段只覆盖最小页面闭环:

- 首页
- 文章详情页
- 独立页面

### 页面与文章分流规则

为保证迁移逻辑可测试且可重复执行, 本阶段采用显式分流规则:

- 位于 `Pages/` 目录下的内容默认判定为独立页面
- 不位于 `Pages/` 目录下的 Markdown 默认判定为普通文章
- 独立页面迁移到 `source/<page>/index.md` 形式, 例如 `Pages/about.md` -> `source/about/index.md`
- 普通文章迁移到 `source/_posts/` 下
- 若后续发现非 `Pages/` 目录中存在本质上属于页面的特殊内容, 第一阶段不自动推断, 而是记录为人工复核项

该规则优先保证简单、稳定与可验证, 不在第一阶段引入复杂启发式判断。

第一阶段不要求完整落地以下页面:

- 分类页
- 标签页
- 归档页
- 友链增强模块
- 复杂主页视觉特效

这些能力可以建立在基础骨架稳定之后再继续扩展。

## 迁移流程设计

### 总体流程

1. 初始化标准 Hexo 站点骨架
2. 建立迁移规则与对应测试
3. 对现有内容做抽样迁移
4. 验证文章页与独立页面是否渲染正确
5. 固化迁移规范, 为后续全量迁移与主题适配做准备

### 迁移边界

本阶段不采用逐篇手工迁移作为主要策略, 而是优先定义规则与脚本化流程。

原因:

- 减少规则调整后的返工
- 为后续批量迁移提供可重复执行的路径
- 保证文章元数据生成方式一致

### 异常内容处理

以下情况在第一阶段标记为人工复核项, 不要求一次性全部自动完美处理:

- 缺少明确首个标题的 Markdown
- Git 历史异常或无法可靠提取时间的文件
- 文件名或路径不适合作为最终 slug 的文章
- 页面与文章边界不清晰的特殊内容

## 输出物与路径规则

### 抽样迁移交付物

第一阶段的抽样迁移验证必须产生可直接被 Hexo 使用的实际文件, 而不是只生成中间结果。

最小交付物包括:

- 抽样文章输出到 `source/_posts/*.md`
- `about` 页面输出到 `source/about/index.md`
- `friends` 页面输出到 `source/friends/index.md`
- 一份记录抽样来源文件与目标路径映射关系的迁移说明, 建议路径为 `docs/migration/hexo-sample-mapping.md`

如需在实现过程中引入中间文件或临时产物, 只能作为内部过程使用, 不能替代最终 Hexo 内容文件。

### 路径与 slug 规则

本阶段采用保守路径策略, 避免过早自定义 permalink 规则:

- 普通文章默认使用迁移后的目标文件名作为 Hexo 输入文件名
- 普通文章目标文件名默认沿用原 Markdown 文件名, 仅在存在明显路径冲突或非法命名时再进入人工复核
- 独立页面默认使用页面目录名作为访问路径, 例如 `source/about/index.md` 对应 `/about/`
- 文章 permalink 在第一阶段优先沿用 Hexo 默认规则, 不额外设计复杂自定义路由
- 如果某篇文章的文件名不适合作为稳定 slug, 则记录为人工复核项, 不在第一阶段强行自动修正

这样可以将“站点先跑通”放在“URL 完美设计”之前, 降低首阶段复杂度。

## 架构边界

为避免后续耦合过重, 第一阶段需要明确分离以下职责:

- Hexo 站点骨架: 负责运行、构建、配置与基础目录
- 迁移规则模块: 负责从旧内容中提取元数据并输出 Hexo 兼容内容
- 页面与文章分流逻辑: 负责判断内容是文章还是独立页面
- 主题适配层: 预留接口, 但不在本阶段深度实现

该拆分可以保证后续主题变更不必重写迁移规则, 迁移规则升级也不应破坏 Hexo 站点基础结构。

## 测试与验证策略

项目要求采用 TDD, 因此第一阶段优先为迁移规则建立测试。

### 测试重点

- 标题提取逻辑是否正确读取首个标题
- Git 时间提取逻辑是否正确生成 `date` 与 `updated`
- 目录到 `categories` 的映射是否符合预期
- 页面与文章的分流逻辑是否正确

### Git 时间提取口径

为避免迁移结果不一致, 第一阶段对 Git 时间提取采用固定口径:

- `date` 使用文件历史中的首次 author date
- `updated` 使用文件历史中的最后一次 author date
- 时间提取以当前仓库中文件路径的 Git 历史为准
- 第一阶段默认不额外处理复杂重命名追踪策略, 若历史因重命名导致无法可靠提取, 记录为人工复核项
- 如果某文件缺失 Git 历史, 不伪造时间, 而是进入人工复核列表

选择 author date 而非 committer date 的原因是它更接近文章创作与修改时间, 更符合博客语义。

### 运行验证

站点级验证至少应覆盖以下命令闭环:

- `hexo clean`
- `hexo g`
- `hexo s`

### 抽样验证范围

抽样内容至少覆盖:

- 2 至 3 篇普通文章
- `about` 页面
- `friends` 页面
- 至少 1 个带明显栏目语义的目录样本

### 抽样迁移完成判定

当以下条件同时满足时, 可认为抽样迁移验证完成:

- 抽样内容已经写入预期 Hexo 目标路径
- 文章与页面都带有可被 Hexo 识别的 front matter
- 首页能够列出抽样文章
- 抽样文章详情页可访问并显示 `title`、`date`、`updated`
- `about` 与 `friends` 页面可通过各自路径访问
- 抽样文章的 `categories` 与旧目录语义一致
- 人工复核项清单已经输出, 可供后续批量迁移继续处理

## 成功标准

当以下条件成立时, 可认为第一阶段完成:

- 本地能够成功执行 Hexo 生成与预览命令
- 抽样迁移后的文章在页面中可正常显示标题、时间与内容
- 独立页面可以按 Hexo 页面模型正常访问
- 现有栏目语义已通过 `categories` 被保留
- 已形成清晰、可重复执行的迁移规则

## 风险与缓解

### 风险: 旧内容格式差异过大, 导致迁移规则失效

缓解方式:

- 先抽样再扩展
- 将异常文件单独记录为人工复核项

### 风险: 过早追求视觉还原, 拖慢基础骨架稳定

缓解方式:

- 第一阶段只保留主题适配接口
- 不把 Akina 风格、`typewriterjs`、自定义 CSS 作为首阶段交付物
- 将视觉细节放到后续阶段

### 风险: 目录迁移后丢失原有语义

缓解方式:

- 强制将旧栏目映射到 `categories`
- 在抽样迁移阶段核对栏目语义是否保留

## 下一阶段计划输入

implementation plan 应重点覆盖以下内容:

1. 初始化 Hexo 站点结构与基础依赖
2. 定义文章与页面的迁移规则
3. 为迁移规则编写测试
4. 抽样迁移少量现有内容并验证
5. 预留后续主题迁移与样式接入点

## 当前结论

本仓库第一阶段应先完成 Hexo 基础骨架搭建, 而不是并行推进动态流、Bot、CI/CD 或完整视觉还原。
在该阶段中, 以标准 Hexo 结构为最终形态, 以规则化迁移保留现有内容语义, 以抽样验证确保后续工作建立在稳定基础之上。
